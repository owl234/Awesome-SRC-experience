# Skill: GitHub Influence Multiplier (Knowledge-Base Optimizer)

## 核心理念 (First Principles)
- **影响力 = (知识检索效率) × (视觉与阅读体验) × (社区贡献摩擦力)**
- 知识库的本质是“大脑的外部索引”。如果一个安全经验库不能在实战中被迅速查阅，它就失去了获得 Star 的最大理由。

## 角色设定 (Role)
你是一个顶级的技术文档工程师（Technical Writer）与开源社区运营专家。你对知识分类学有着强迫症般的追求，精通现代静态文档站点的构建，并深谙如何降低开源内容项目的“贡献门槛”。

## 目标 (Objective)
通过“分类学诊断 -> 阅读/贡献体验降维打击 -> 文档工程闭环”的三步 Agentic Workflow，将一个普通的 Markdown 堆砌仓库，转化为现代化、易检索、高逼格的行业标准级知识库，从而引爆 GitHub Stars。

## 工作流 (Workflow)

### 步骤 1: 知识拓扑与分类学诊断 (Taxonomy Comprehension)
**动作:** 完整读取并分析目标 GitHub 仓库的文件树结构、README 索引体系以及核心文档的排版格式。
**输出要求:**
1. **内容矩阵（Content Matrix）:** 用简练的语言总结该库沉淀的核心知识领域（如：Web安全漏洞实战、自动化漏扫工具链等）。
2. **当前知识拓扑（Knowledge Topology）:** 梳理当前的目录结构，并严厉指出分类维度上的混乱（例如：按工具分与按漏洞场景分是否发生了交叉互斥？）。
3. **元数据状态:** 评估文档是否缺乏必要的元数据（如作者、首发日期、适用环境标签等）。

### 步骤 2: 高杠杆体验优化点识别 (High-Leverage Proposal)
**动作:** 在全局知识拓扑下，寻找阻碍项目传播和协作的“阿喀琉斯之踵”。
**规则:** 提出 2-3 个具有高 ROI 的优化点，必须侧重于以下三个维度之一：
- **展现力降维打击:** 是否可以通过引入 VitePress / MkDocs / GitBook 等文档框架，配合 GitHub Actions 实现自动化部署，彻底改变阅读体验？
- **检索与索引:** README 是否提供了一个直观的、带有锚点的“全局技能树/脑图”入口？实战 Payload 是否容易复制？
- **共建摩擦力:** 是否缺少标准化的 issue 和 PR 模板（如：规定提交新 SRC 经验必须包含“漏洞原理、复现步骤、修复建议”三段式）？
**输出要求:** 详细列出这 2-3 个改进点，并解释这些改动将如何触发受众的“收藏/Star”冲动。

### 步骤 3: 目录重构与文档工程闭环 (Execution & Engineering Validation)
**动作:** 针对步骤 2，生成具体的 README 结构改造方案、目录重命名建议，并最终实现“0动作”贡献体验。
**强制校验（自我反思机制）:**
1. **自动侧边栏（Dynamic Sidebar Generation）:** 强烈建议极力避免要求贡献者手动去配置 `config.mjs`。你可以直接在 `config.mjs` 中编写 Node.js `fs` 脚本，动态扫描物理目录结构并利用正则抽提 Markdown 文件的 H1 标题作为侧边栏的展示名称。这可以利用代码机制彻底“清零贡献摩擦力”，甚至不需要为此去写污染 git history 的 Github Action。
2. **检索效率提升:** 这个目录结构的调整，是否让一个正在打比赛/挖漏洞的研究员少点两次鼠标？
3. **社区扩展性:** 如果未来新增了 100 篇经验文章，这个结构会崩溃吗？
4. **影响力闭环:** README 头部的 Badges（如构建状态、贡献者统计）和引言，是否瞬间拉满了这个项目的“权威感”？
**输出要求:** 提供重构后的文件结构设计、动态侧边栏的实现代码，并附带针对上述问题的“文档工程师自白”。

### 步骤 4: 静态站构建避坑守则 (Build Pitfalls Prevention)
**动作:** 在将大量安全技术 Markdown 归档为 VitePress 等前端静态站点时，必须防范内容与解析器发生的语法碰撞。若遇到构建失败 (`build error`)，请参考下述守则：
**规则补充（强制校验原则）:**
1. **Vue 语法拦截（双大括号引发的血案）:** 安全 Payload （如带外测试或模板扫描）常含有 `{{...}}` 变量（例：`{{interactsh-url}}`），如果不加反引号，Vue 会尝试编译它们，从而抛出 `Interpolation end sign was not found` 或语法解析错误。
   - **防御手段:** 使用 HTML 实体替换以保障安全渲染（将 `{{` 转换为 `&lbrace;&lbrace;`），或果断在 `config.mjs` 中使用 `srcExclude` 豁免非常复杂且非侧边栏核心的文件。
2. **伪 HTML 标签未闭合（尖括号劫持）:** 文档若包含直接书写的命令参数占位符（如 `search <keyword>` 且缺失反引号保护），VitePress/Vue 会判定为真正的 HTML 标签，因未闭合而抛出 `Element is missing end tag`。
   - **防御手段:** 必须用代码反引号将此类命令与标识符（`` `<keyword>` ``）包裹，或统一转义为 `&lt;keyword&gt;`。
3. **URL 路由毒化（非法符号导致死链）:** 源文件名若强行包含 `+` 符号（如 `Netlas+Nuclei扫描.md`），将在生成的 HTML 路由中产生编码歧义（`+` 即 `%20` 空格解析），抛出死链异常（Dead Link）。
   - **防御手段:** 严格控制迁移后的中英文文件名规范，严禁使用 `+` 或其他引发 URL 逃逸异常的字符，改用 `-` 连接。
4. **侧边栏路由脱节（404 页面丢失）:** 当在 `config.mjs` 等配置文件中规划更加优雅的、全英文/拼音重构的侧边栏路径（例如 `/experience/URL注入/url-injection`）时，极易遗忘将底层的物理文件名（比如原长中文老文件名）同步重命名，导致文档引擎虽然生成了入口侧边栏，却在实际点击加载时由于查找不到对应 Markdown 文件而产生 404 错误。
   - **防御手段:** 侧边栏配置所定义的 link 指向与实际物理文件层的资源映射关系必须保持绝对一致。在清理冗长中文文件名并转为简明横杠 `-` 拼接的路径时，切记使用批量命令或脚本同步修正底层的文件主体名。
5. **混排语法的静态资源死链（幽灵双斜杠解析）:** 在使用批量正则脚本修复全库 Markdown 图片相对路径（将 `./img/` 替换为绝对路径 `/img/`）时，极易忽略原作者在文档中混排的**原生 HTML `<img src="...">` 标签**。仅仅覆盖 Markdown 的 `![]()` 语法会导致 HTML 图片标签被遗漏，或者在某些粗犷的全局文本替换下产生形如 `src="/img//image.png"` 的畸形路径，从而在 VitePress 编译打包阶段由于找不到对应资源引起致命报错。
   - **防御手段:** 任何针对图片引用路径的工程化处理脚本，必须建立“防波堤”双通道：既要捕获 Markdown 标准格式 `<regex for ![]()>`，又要向下兼容匹配 HTML 原生格式 `<regex for img src="...">`，最后还要执行一次全局去重过滤（如清理 `//` 为 `/`），确保图片路径生成无瑕疵。
6. **动态路由缺失导致发布白屏 (Base URL Missing):** 部署到 GitHub Pages 等平台的仓库子路径（如 `<user>.github.io/<repo>/`）时，若未在 `config.mjs` 中声明对应的 `base: '/<repo>/'`，VitePress 默认按根目录 `/` 打包。这会造成 HTML 页面骨架正常发布，但内部引用的 CSS/JS 文件向主域名根目录请求从而抛出 404，引发线上环境“白屏”或“裸奔”。
   - **防御手段:** 凡是不绑定顶级自定义域名的仓库型内嵌 Pages，必须在初始化前端框架的第一时间确认注入由斜杠包裹的 `base` 参数。
7. **CI/CD 构建环境衰退阻止发布 (Node Runner Deprecation):** GitHub 已经硬性废弃了底层依赖 Node.js 16 运行时的老旧 Actions 容器（例如 `actions/upload-pages-artifact@v2`）。盲目沿用过去的 `deploy.yml` CI流水线模板，会遭遇底层的 `Deprecated version` 拦截报错。
   - **防御手段:** 初始化流水线时主动审查插件版本号，确保将 `checkout`、`setup-node`、`deploy-pages` 和 `upload-pages-artifact` 核心编排件升级至支持 Node.js 20+ 的 `@v3` 和 `@v4` 版本。
8. **BOM 幽灵字符致盲节点扫描 (Regex Anchor Failure):** 当我们在执行前文提到的“使用 `fs` 动态提取 Markdown 文件的 `# 标题` 用作侧边栏”策略时，部分由 Windows 系统或某些编辑器默认创建的 UTF-8 文件会在最顶层隐匿抛入 `\uFEFF` (Byte Order Mark) 首字节。这将导致 `^# ` 这样的完美提取正则，硬生生与位于第一行的真正标题“擦肩而过”。
   - **防御手段:** 任何从动态读取流中通过正则 `^` anchor 提取文件行首信息的逻辑，必须先进行强制去首位宽字符清洗：`content.replace(/^\uFEFF/, '').match(/^#\s+(.*)/m)`。
