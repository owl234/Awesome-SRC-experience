# Awesome SRC Experience - 项目定位与 Agent 指南

在协助处理本项目 (Awesome SRC Experience) 的任务时，所有的 AI Agents 都必须严格遵守以下项目核心定位与设计原则：

## 1. 强制计划与审批 (Mandatory Planning & Approval) - 【最高优先级】
- **核心理念**：在执行任何代码修改或文件变更前，必须先与用户对齐方案。
- **具体要求**：每次接受任务时，必须先生成并输出一份清晰的实施计划（Implementation Plan）。只有在用户审阅、修改并明确确认该计划“没问题”之后，才能开始执行具体的代码修改或终端命令运行任务。绝对不可未经允许直接擅自修改项目代码。

## 2. 极简美学 (Apple-like Design Philosophy)
- **核心理念**：参考苹果 (Apple) 的设计理念，始终追求**简约、美观、克制**。
- **具体要求**：在进行前端页面修改、UI/UX 调整或文档排版时，避免任何冗余的装饰。注重留白、排版结构和极致的阅读体验。如果需要添加新组件或页面，务必保持视觉上的高品质与现代化。

## 3. 极简的内容维护流 (Simplicity in Updates)
- **核心理念**：更新或添加知识库内容的过程必须极其简单。
- **具体要求**：在编写代码或调整项目架构时，绝对不能引入繁琐的配置文件或复杂的构建步骤。必须维护“零配置索引”的理念，确保用户只需将 Markdown 文件放入对应目录，即可自动完成所有内容的渲染和侧边栏的更新。

## 4. 降低参与门槛 (Low Barrier to Entry)
- **核心理念**：让更多的人能更快地熟悉本项目，大幅降低社区参与门槛。
- **具体要求**：无论是代码结构还是文档说明，都应当直白易懂。在生成贡献指南 (Contributing)、Issue 模板或自动化脚本时，要站在新手的角度思考，提供最友好的指引，让极客共建变得轻而易举。

## 5. 技术栈与框架约束 (Tech Stack Constraints)
- **核心理念**：维持项目的轻量级架构，拒绝过度工程化。
- **具体要求**：本项目是一个纯静态文档站点，基于 VitePress 构建。在提供代码建议时，**请勿**引入任何需要后端服务器运行的语言（如 Python、Node.js 服务端代码）或数据库。前端交互请仅使用原生的 HTML/JS/CSS 或 VitePress 支持的轻量级 Vue 语法，避免引入 React 等其他重型框架。

## 6. 文档排版与格式规范 (Markdown Formatting Rules)
- **核心理念**：提供统一且高质量的文字阅读体验。
- **具体要求**：在生成或修改 Markdown 文档时，请严格遵守中文排版规范（如中英文、数字之间保留一个半角空格）。代码块必须标明具体的编程语言，所有的外部链接必须检查有效性。使用统一的标题层级（H1 仅用作页面标题，内容从 H2 开始）。

## 7. Git 提交规范 (Commit Message Guidelines)
- **核心理念**：保持版本控制记录的专业性和可读性。
- **具体要求**：在协助提交代码或生成 Commit Message 时，请严格遵守 Conventional Commits 规范（如使用 `feat:`, `fix:`, `docs:`, `style:` 等前缀），并保持提交信息的简明扼要，方便开源社区的代码审查。

## 8. 语言与语气规范 (Language & Tone)
- **核心理念**：专业、客观、友好的社区氛围。
- **具体要求**：本项目的核心受众是中文安全圈（尤其是 SRC 白帽子）。在编写指南、Issue 回复或文档内容时，请使用专业、克制且友好的中文。对于常见的安全术语（如 XSS, SSRF, RCE）可保持英文全大写，无需过度翻译。
